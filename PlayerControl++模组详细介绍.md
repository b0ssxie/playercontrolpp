## 1. 模组概述

PlayerControl++ 是一个纯客户端的 Fabric 模组，旨在增强原版玩家的操作体验。所有功能均通过模拟玩家按键输入实现，不直接修改实体位置或发送异常数据包，尽可能兼容多人服务器与反作弊环境。

**核心功能一览：**

| 功能 | 描述 |
|------|------|
| 自动前进 | Toggle 热键持续前进，模拟按住 W 键 |
| 快速转向 | 按键瞬间旋转指定角度（0-360°） |
| 路径流系统 | 预设多导航点路径并自动行走 |
| 录制与回放 | Tick 级输入录制，RLE 压缩，NBT 二进制格式保存 |
| 自动缓存容器 | 扫描并自动打开附近容器供 ChestTracker 缓存 |
| 自动注水（v1.4 新增） | 扫描投影可含水方块并自动使用水桶含水 |
| 自动材料收集 | 基于 Baritone + ChestTracker 从投影收集材料 |

**代码架构：** 约 10000+ 行 Java 代码分布在 47 个源文件中，采用模块化设计，各功能相互独立，通过事件系统和反射调用实现联动。

## 2. 路径流系统

路径流系统是模组的核心功能之一，允许玩家预设多条带导航点的路径并自动行走。

### 2.1 路径管理

- 每条路径支持多个中间导航点（起点 → 途经点 → 终点）。
- 支持添加 / 删除路径、添加 / 删除导航点、动态热键绑定。
- 路径数据以 JSON 格式持久化存储于 `.minecraft/config/` 目录。
- 路径名称可自定义，支持中英文。

### 2.2 路径执行

- 模拟按住 W 键前进 + 自动 yaw 视角修正，模拟真人行走行为。
- **混合转向模式**：允许玩家轻微移动视角，偏差过大时自动平滑修正。
  - 分级修正速度：偏差 >45° 时 25°/tick，>15° 时 18°/tick，否则默认 15°/tick。
  - 死区 ±2° 内不修正，避免"锁头"感。
- 仅控制 XZ 平面，忽略垂直偏差（Pitch）。

### 2.3 循环模式

| Loop Count | 行为 |
|------------|------|
| 1 | 单次：起点 → 终点，结束 |
| >1 | 往返 N 次：起点 → 终点 → 起点（往复循环） |
| 0 | 无限循环 |

### 2.4 卡住检测

- 连续 3 秒未检测到位移 → 自动触发一次跳跃。
- 跳跃后 5 秒内仍未移动 → 终止当前路径，ActionBar 提示"路径终止：卡住"。

### 2.5 安全系统

- 世界切换、维度变更（下界↔主世界↔末地）、玩家死亡、从服务器断开 → 立即停止路径执行。
- 配置界面打开时暂停路径执行，关闭后恢复。

### 2.6 路径级别选项

每条路径可独立配置以下选项：

| 选项 | 说明 |
|------|------|
| **Sprint（疾跑）** | 开启后执行路径时全程按住疾跑键 |
| **LayerCtrl（图层控制）** | 开启后每次遍历完成自动切换 Litematica 投影渲染层 |

## 3. 自动前进与快速转向

### 3.1 自动前进

- 可自定义 Toggle 热键（默认无绑定）。
- 按下后模拟持续按住 W 键前进。
- 切换世界或维度时自动关闭，ActionBar 显示状态提示。
- 再次按热键或手动按 W/S 键可关闭。
- 轻量实现，不干扰其他按键操作。

### 3.2 快速转向

- 可自定义 Trigger 热键（默认无绑定）。
- 按下瞬间旋转指定角度，默认 180°，范围 0-360°。
- 角度可在配置 GUI 的 Settings 标签页中调整。

## 4. 玩家行为录制与回放

以 Tick 级精度录制玩家输入并回放，v1.4 进行了重大重构，从 JSON 格式迁移至 NBT 二进制格式。

### 4.1 录制数据内容

| 数据类型 | 说明 |
|----------|------|
| 移动 | WASD 四个方向键状态 |
| 疾跑 | Sprint 键状态 |
| 跳跃 | Jump 键状态 |
| 潜行 | Sneak 键状态 |
| 视角 | Yaw（水平旋转）+ Pitch（垂直旋转） |
| 左键 | 攻击/破坏 操作 |
| 右键 | 使用/放置 操作 |
| 位置关键帧 | 每 20 Tick 记录一次玩家坐标 |

### 4.2 RLE 压缩

- 连续 Tick 中完全相同的输入状态合并为一个 Segment（段）。
- 每段包含 `duration` 字段表示该段持续的 Tick 数。
- 大幅减少录制文件大小，同时保持回放精度与原输入完全一致。

### 4.3 NBT 二进制格式（v1.4 重构）

v1.4 对录制系统进行了以下重大改进：

- **文件格式变更**：录制文件由 JSON（`.json`）改为 NBT 二进制格式（`.pcr` 扩展名），读写效率更高。
- **RingBuffer 录制缓冲**：录制过程中使用分块预分配数组（CHUNK_SIZE=1024），大幅减少 GC（垃圾回收）压力，录制更流畅。
- **后台异步写入**：录制停止后，数据在独立后台线程（线程名 `PCpp-RecSave`）写入 `.pcr` 文件，不阻塞主线程，避免录制结束时的卡顿。
- **始终记录位置关键帧**：移除了低精度模式，所有录制始终每 20 Tick 记录位置关键帧（PositionKeyframe），回放时在关键帧 Tick 检查位置偏差 >0.2 块时自动修正至录制轨迹。
- **损坏文件检测**：加载异常的 `.pcr` 文件时在 ActionBar 显示错误提示，避免读取无效数据导致崩溃。

### 4.4 回放功能

- Play Count = 0 无限循环回放。
- Play Count = N 重复回放 N 次。
- 回放开始 / 每次循环开始时，模拟行走至录制起点位置（不使用瞬移），保持操作真实性。
- 按需加载录制文件：录制列表 GUI 仅读取索引文件，用户点击"播放"时才读取完整录制数据，减少内存占用。
- 回放过程中自动修正位置偏差（基于位置关键帧比对）。

### 4.5 文件存储结构

```
.minecraft/
└── config/
    └── playercontrolpp/
        └── recordings/
            ├── index.json          # 录制索引文件（名称、日期、时长等元数据）
            ├── rec_001.pcr         # 录制文件（NBT 二进制格式）
            ├── rec_002.pcr
            └── ...
```

- **索引文件**（`index.json`）：存储所有录制的元数据（名称、创建时间、Tick 时长等），GUI 录制列表仅读取此文件。
- **录制文件**（`.pcr`）：每个录制独立保存为 NBT 二进制文件。
- 删除操作仅删除对应 `.pcr` 文件并更新 `index.json`。

### 4.6 安全机制

- 播放期间禁止开始新录制。
- 录制 / 播放开始时自动退出所有 GUI 界面。
- 录制期间 ActionBar 常驻"正在录制"提示。
- 世界切换、死亡、断开连接时自动停止录制和回放。

## 5. Litematica 投影联动

路径执行过程中，每次完成一次遍历（起点→终点 或 终点→起点），自动切换 Litematica 投影的 SINGLE_LAYER 渲染层。

**实现细节：**

- 通过 Java 反射调用 Litematica API，无编译时依赖。
- Litematica 未安装时静默降级，不影响其他功能正常使用。
- 仅在 SINGLE_LAYER 渲染模式下生效（非 SINGLE_LAYER 模式跳过）。
- 支持自定义图层增量（正数向上层 / 负数向下层，默认 +1）。

## 6. 自动缓存附近容器

帮助 ChestTracker 等容器缓存类模组自动获取附近容器内容，无需手动逐个打开容器。

### 6.1 扫描机制

- 以玩家位置为中心、当前游戏模式最大交互距离为半径，球形范围扫描。
- 通过 ChestTracker 反射 API 获取已缓存容器列表，仅处理未缓存容器。
- 按距离从近到远排序，优先打开距离最近的容器。
- 支持自定义可缓存容器白名单，默认包含全部 28 种原版存储容器（箱子、木桶、潜影盒、漏斗、发射器、投掷器等）。

### 6.2 状态机（6 状态）

```
SCANNING → OPENING_CONTAINER → WAITING_AFTER_OPEN → CLOSING_GUI → COOLDOWN → AUTO_STOP_COUNTDOWN
```

| 状态 | 说明 |
|------|------|
| SCANNING | 扫描附近容器，发现未缓存容器进入下一状态 |
| OPENING_CONTAINER | 模拟右键交互打开容器 |
| WAITING_AFTER_OPEN | 等待容器 GUI 完全加载，确保 ChestTracker 完成缓存 |
| CLOSING_GUI | 关闭容器 GUI |
| COOLDOWN（v1.4 新增） | 关闭容器后延迟等待（可配置 1-200 Tick），首个容器无延迟 |
| AUTO_STOP_COUNTDOWN | 3 秒内无新容器可处理则自动停止 |

### 6.3 缓存容器延迟（v1.4 新增）

- 新增"缓存容器延迟"设置项，范围 1-200 Tick，默认 1 Tick。
- COOLDOWN 状态在关闭当前容器后、打开下一个容器前生效。
- 第一个待处理容器立即打开，无延迟。
- 3 秒自动停止倒计时不受延迟设置影响，在 COOLDOWN 期间继续倒计时。

### 6.4 动态重扫描

- 玩家移动时持续重新扫描范围内容器。
- 新进入扫描范围的未缓存容器自动加入处理流程。
- 已缓存的容器不重复打开。

### 6.5 安全与兼容性

- 所有操作通过正常客户端交互流程完成（模拟右键 → 等待 → 关闭 GUI）。
- 兼容单人游戏、局域网和多人服务器。
- 世界切换时立即停止所有操作。

## 7. 自动注水

v1.4 新增功能，自动检测 Litematica 投影中的可含水方块并使用水桶进行含水操作。

### 7.1 功能概述

- 需要 Litematica 投影（schematic）已加载。
- 扫描投影中玩家可达距离内的可含水方块（waterloggable blocks）。
- 自动在快捷栏 / 背包中寻找水桶并切换至手持。
- 自动旋转视角面朝目标方块，模拟右键进行含水操作。
- 所有操作通过 `interactBlock` + `interactItem` 完成，兼容服务器端含水逻辑。

### 7.2 状态机（6 状态）

```
IDLE → SCANNING → FINDING_WATER_BUCKET → ROTATING → WATERLOGGING → COOLDOWN
```

| 状态 | 说明 |
|------|------|
| IDLE | 等待启动 |
| SCANNING | 在投影中扫描可含水方块 |
| FINDING_WATER_BUCKET | 在背包中寻找水桶并切换至手持 |
| ROTATING | 平滑旋转视角至目标方块 |
| WATERLOGGING | 执行 interactBlock + interactItem 右键含水 |
| COOLDOWN | 操作延迟等待后继续扫描下一个 |

### 7.3 配置项

| 设置 | 范围 | 默认值 | 说明 |
|------|------|--------|------|
| 注水扫描半径 | 0-5 格 | 3 格 | 以玩家为中心扫描可含水方块的半径 |
| 注水操作延迟 | 1-200 Tick | 10 Tick | 每次含水操作后的冷却间隔 |

### 7.4 热键

- 新增热键"自动注水"（Auto Water Fill），默认无绑定，需在配置 GUI 中手动设置。
- 按下 Toggle 开启/关闭。

### 7.5 安全机制

- 潜行（Sneak）时暂停操作。
- 玩家死亡时自动停止。
- 世界切换时自动停止。
- 水桶用完时自动停止。

## 8. 自动材料收集

基于 Baritone + ChestTracker 从 Litematica 投影自动收集所需材料。

### 8.1 功能概述

- 需要 Baritone 和 ChestTracker 模组。
- 读取 Litematica 投影所需的材料列表。
- 通过 ChestTracker 查询附近已缓存容器中是否有所需材料。
- 调用 Baritone 自动移动到容器位置。
- 背包满时自动将多余物品存入潜影盒（需 QuickShulker 联动）。

### 8.2 配置

- 在配置 GUI 的 **Baritone** 标签页中进行相关设置。
- 支持配置每类材料的目标收集数量。

## 9. 配置 GUI 标签页

配置界面通过默认热键 `P + C` 打开，也可从 Mod Menu 进入。分为以下标签页：

| 标签页 | 内容 |
|--------|------|
| **Hotkeys** | 通用热键绑定：打开配置界面、自动前进、快速转向、切换录制、自动缓存附近容器、自动注水 + 可缓存容器白名单 |

| **Recording** | 录制与回放管理（录制列表 / 播放控制 / 损坏文件检测） |
| **Baritone** | 自动材料收集配置 |

## 11. 安全与兼容性

### 11.1 设计原则

- **纯客户端**：不安装服务端版本，不修改服务端逻辑。
- **模拟输入**：所有操作通过模拟玩家按键和交互实现，不发送异常数据包。
- **反射调用**：所有联动模组通过反射调用，无硬依赖。

### 11.2 反作弊兼容性

- 不直接修改实体位置（teleport）。
- 不修改移动速度或飞行状态。
- 模拟正常玩家输入时序。
- 已在多个主流服务器测试通过，但无法保证所有反作弊插件兼容。

### 11.3 安全停止条件

以下事件会触发所有自动化功能的立即停止：
- 世界切换 / 维度变更
- 玩家死亡
- 从服务器断开连接
- 潜行（仅暂停自动注水）
